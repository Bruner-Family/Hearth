# Hearth v1.5 "Reference Details" Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Each item gains an ordered key-value "reference details" list (e.g. `Filter: 16×25×1`, `Bulb: A19 60W`, `Paint: SW 7029`) — editable on the item form, shown prominently on item detail, and matched by the existing fuzzy search — so the spec you need standing at the appliance or in the store aisle is one screen away.

**Architecture:** A single `jsonb` column `reference_details` on `items` holds an ordered array of `{ label, value }` pairs (a handful per item doesn't justify a table + RLS — roadmap §3 v1.5). A pure helper (`cleanReferenceDetails`) trims and drops blank rows; the v1.4 `searchItems` gains a Fuse `getFn` key so labels and values are searchable. The item form edits the list with a react-hook-form `useFieldArray`; item detail renders a read-only card. Everything else (RLS, grants, types) rides on the existing `items` machinery.

**Tech Stack:** Supabase Postgres (migration + pgTAP), hand-maintained TS types, Fuse.js (existing), react-hook-form `useFieldArray` + zod, Expo/React Native Web + NativeWind, vitest.

**Spec:** `docs/superpowers/specs/2026-06-11-hearth-feature-roadmap-design.md` §3 v1.5 (the "Reference details" bullet). This is the **first of four** independent v1.5 "At the Appliance" sub-features; document attachments, QR labels, and CSV import each get their own plan.

**Decisions baked in:**
- **Storage shape: ordered array of `{ label, value }`** (not an object map) — preserves the order the user entered, allows repeated labels, and maps 1:1 to the editable list UI. JSONB column with `default '[]'` and a `jsonb_typeof = 'array'` check; per-element shape is validated in the app (lenient: blank rows are dropped on save, never error).
- **Searchable** via a Fuse `getFn` key indexing both labels and values (weight 1, same as notes).
- **No CSV-export column** for reference details (v1.4 export has fixed columns; a variable-length list doesn't fit and isn't in scope). 

**Branch:** create `claude/v1.5-reference-details` from `main` (v1.4 is merged).

## Codebase orientation (read once before starting)

- **Commands:** `npm run typecheck` (tsc), `npm run lint` (expo lint — **CI is zero-warnings strict**), `npm test` (vitest, runs once). SQL: `supabase db reset` (re-applies migrations), `supabase test db` (pgTAP) — these need a local Supabase (`supabase db start`); if Docker/Supabase isn't available in your environment, write the SQL correctly and rely on review + the JS gates, noting the pgTAP wasn't run.
- **Test scope:** vitest covers **pure `src/lib/*.ts`** only — its `include` is `src/**/*.test.ts`. Tested modules must not import `react-native`. Components (`.tsx`) are verified by typecheck + lint (+ manual), not unit tests. Test fixtures live in `src/lib/__tests__/`; `makeItem(overrides?)` builds an `ItemWithCategory`.
- **DB conventions:** multi-tenancy by `household_id`; **RLS is the only authz layer** and already governs `items` (`supabase/migrations/20260610000002_rls_policies.sql`). Table grants are table-level (`20260610000005_grants.sql`) so a new column on `items` needs **no grant change**. `private.set_updated_at()` trigger already fires on `items` update. pgTAP pattern: copy the `test_as(uid, email)` helper and the new-user-auto-gets-a-household assumption from `supabase/tests/notifications.test.sql`.
- **Types:** `src/lib/database.types.ts` is hand-maintained; keep it in lockstep with migrations. `TableOf<Row, Required, Generated>` builds Row/Insert/Update — a column with a DB default belongs in neither `Required` nor `Generated`, so it lands in the `Partial` (optional on Insert). 
- **Forms:** react-hook-form + `zodResolver`; schemas in `src/lib/schemas.ts`. Use `Controller` for inputs and `useFieldArray` for the list. **Never** call `watch()` (trips `react-hooks/incompatible-library` under `--max-warnings 0`); this feature doesn't need it.
- **Item form** (`src/components/ItemForm.tsx`): a `useForm<ItemFormValues>` with `Controller`-wrapped `Field`s; on submit it maps form values → `Omit<ItemInsert, "household_id">` and calls `onSubmit`. The category picker shows the established **chip** pattern.
- **Item detail** (`src/app/(app)/items/[id]/index.tsx`): a `ScrollView` of `Card`s; a "facts" `Card` lists label/value rows (`flex-row justify-between`, `border-t border-edge` between rows). New reference card mirrors that style.
- **UI kit** (`src/components/ui.tsx`): `Card`, `SectionTitle`, `Button` (variants primary/secondary/danger), `Field` (labeled TextInput), `ErrorNote`. Raw colors via `usePalette()` (`@/lib/theme`) — only for `placeholderTextColor` and the like.
- **Search** (`src/lib/search.ts`): `searchItems(items, query)` builds a Fuse index over `FUSE_OPTIONS.keys`; a key may be `{ name, weight, getFn }` where `getFn: (item) => string | string[]`.
- **Demo mode** (`src/lib/demo.tsx`): `seedItem(...)` builds demo items; `demoDb.createItem/updateItem` must carry any new column so the form round-trips in the offline demo.

```ts
import type { ReferenceDetail } from "@/lib/database.types";

/**
 * Normalize the form's reference-detail rows for storage: trim each field and
 * drop any row missing a label or value (so blank/half-filled rows are silently
 * discarded rather than erroring). Order is preserved.
 */
export function cleanReferenceDetails(
  pairs: ReferenceDetail[],
): ReferenceDetail[] {
  return pairs
    .map((p) => ({ label: p.label.trim(), value: p.value.trim() }))
    .filter((p) => p.label !== "" && p.value !== "");
}
```
